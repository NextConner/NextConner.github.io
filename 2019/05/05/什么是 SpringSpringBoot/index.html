什么是 Spring/SpringBoot?

  • 概念 :

    • 从广义上理解：

      • Spring不是单指 Spring 应用开发框架，它指的是 Spring 一整个的JEE开发平台，包括

        • Spring Initializr 脚手架工具。
        • Spring Boot 快速搭建配置框架。
        • Spring Framework 本身包含的 Spring MVC 和 Spring WebFlux 框架。
        • Spring Data Umbrella 项目合集，包含了 : Spring Data JPA，Spring Data JDBC，Spring Data MongoDB。
        • Spring Cloud Umbrella 项目合集,Netfix 开源的微服务项目
        • Spring Cloud Data Flow ，集成了 Spring Integration 和 Spring Batch 项目
        • Spring Security等

      • 主要是以上这些大的部分，构成了Spring 一整套的JEE开发平台

  • 从狭义上理解:

    • Spring 指的就是Spring MVC Framework,特别是 Spring 框架中一些核心的概念和思想，比如依赖注入，切面编程，数据绑定这些。

      • IOC : 依赖注入：

        概念:

        • 指的是由 创建对象所需要的依赖 转变为提供 对这些依赖 是什么，以及如何创建 的描述，直到真正需要创建依赖对象时，

        通过反射机制去动态地创建对象。

        什么是反射/JAVA 的反射机制:

        • 非侵入式编程

        • 依赖注入过程:

          • 初始化 IOC 容器 :

            IOC 容器的实现 :

            ​ 1.ApplicationContext (最常见也是功能最丰富的 IOC 容器)

            ​ 2.XmlBeanFactory

          • 定位 Bean 配置文件

          • 加载并解析配置文件，这里会获取到Bean 的属性，也就是 标签的值

          • 将解析后的 Bean 信息创建为 BeanDefinition 对象，存在 BeanDefinitionMap 中

          • 注册 BeanDefinition

            依赖注入的发生:

            • 当容器调用 getBean() 方法索要 Bean 对象时候，AbstractAutowireCapableBeanFactory 创建 Bean 实例对象
            • createBeanInstance 方法创建 Bean 的 Java 实例对象

        • AOP : 切面编程

          • 为了组织与核心业务代码无关但是为业务代码服务的功能点，切面编程产生了

          • 它将程序中的次要关注点例如日志，权限验证等组织在别的地方，并且定义何时在何处(通常是 切点–>即业务方法，调用前后)执行

            这些切面代码。

          • 不会污染核心业务代码，关注的是纵向的逻辑。

            切面 : 由横切关注点模块化而成的类就是切面；

            使用切面有什么好处 :

            • 每个关注点都集中在一个地方，而不是分散在代码中，便于统一管理
            • 核心业务代码更加简洁，因为它们只需要关注业务，次要关注点都被转移到切面中

        • 依赖注入的方式:

          • 在 java 中有三种方式实现依赖注入:

            • 构造器注入:
            • Setter 方法注入
            • 接口注入

          • Spring 提供几种方式装配Bean:

            • 自动装配 Bean : 基于两个角度来实现

              • 组件扫描(Component Scan) : Spring 会自动发现 Bean 容器 中所创建的 Bean

              • @Component(“BeanId”) 注解标记当前类作为 组件类 ， @ComponentScan(BasePackages) 会开启组件扫描(例如指定的包路径)

                • 最佳实践 :

                  • @ComponentScan(value) : value 支持两种格式 : basePackage，指定扫描的包路径；

                BasePackageClasses = {className,…} : 设置为具体的组件类，一个比较好的实践是，使用包中的空标记接口

                来代替具体的类型，这样在代码重构之后，即使移除了特定的实现类型，也不会因为扫描不到组件报错。

                • 在SpringBoot 项目中，通常建议将Application 类放在顶层目录之外，这样一些自动化配置会扫描到所有

                的包路径，例如上文的 @ComponentScan ，不需要额外指定扫描路径

              • 基于Java 依赖注入规范的 @Named(“BeanId”) 方法也可以为Bean 设置ID

              • 自动装配(autowired) : Spring 自动满足 Bean 之间的依赖

              • @Autowired : 自动装配注解

                • 可以作用在构造器，setter方法或者直接作用在域上
              • 遵循Java 依赖注入规范的 @Inject 注解也可以达到依赖注入的目的

            • 通过JAVA 代码装配Bean :

              • @Configuration 注解:

                • 这个注解会将一个 Java 类声明为一个配置类，该类会包含在 Bean 容器中如何创建 Bean 的细节（这和提供BeanDefinition 文件，描述Bean 是一样的行为）

                • 还需要在配置类中添加一个被 @Baen 注解修饰的方法，表示这个方法将会返回一个Spring 容器中的使用的Bean ，默认BeanId 就是带有此注解的方法名，但可以自定义注解的 name 属性，例如 : @Bean(name="lonelyYi~~~~").

                • 那这种情况下如何实现注入:

                  • 借助Java Config 实现 ：

                  • 第一个CDPlayer 方法的实现是将两个Baen 声明在同一个配置类中，和 xml 描述Bean 是一模一样的

                    • 第二个方法就可以把 Bean 声明分散开，有一定的解耦作用，而且还可以在方法体通过setter方法或者域来注入对象

                • 在配置类中导入配置类 :

                  • 例如上例中的 CDPlayer bean 方法，gameBoy bean 如果声明在另一个configuration 类中，就需要在配置类上使用 @Import(Config.class) 注解来引入需要的配置类
                  • 如果 gameBoy 声明在 xml 文件中，需要使用 @ImportResource (path) 注解来代替 @Import 注解。
        - Bean 的作用域: 

          - 主要是四种:

            >Singleton(单例):在整个应用中只创建一个 bean 实例，也就是注入到每个bean 中的依赖bean,是同一个bean对象
            >
            >Prototype(原型):每次通过上下文注入或者通过spring容器获取bean 的时候，都会创建一个新的bean实例
            >
            >Session(会话级别):在`web应用`中，为每一个Session 创建一个 bean 实例
            >
            >Request(请求级别):在 `web应用`中，为每个请求创建一个bean 实例



          - 可以在 @Component 注解修饰和 @Bean 注解修饰的类或者方法上使用 @Socpe 注解来设置Baen 的作用范围

Spring 面试题

  • Spring IoC 容器 : 侧重于容器这个概念，是负责创建，装配，以及管理Bean生命周期的东西。

    • Spring 容器 使用 依赖注入 来管理组成应用程序的bean对象
    • 容器通过读取提供的配置元数据 Bean Definition 来接收对象进行实例化，配置和组装的指令。
    • 该配置元数据 Bean Definition 可以通过 XML ， Java 注解 ，Java Configuration 类提供。

  • 什么是(DI)依赖注入: 你不必主动，手动创建对象，但必须描述如何创建它们。

    • 不是直接在代码中将组件和服务链接在一起，而是在配置文件中描述哪些组件需要哪些服务。
      • 个人理解: 在配置文件或配置类中描述组件的属性和它们之间依赖关系
    • 然后由IoC 容器将它们装配到一起。

  • IoC 和 DI 有什么区别 ：

    • IoC 是一个更加宽泛的概念，DI 更加具体一点，都是表示一种将调用者和被调用者分离的思想，达到解耦的目的

    • DI 在 Spring 有三中具体的实现方式 ：

      • Interface Injection : 接口注入
      • Setter Injection : setter 注入
      • Constructor Injection : 构造注入
- Spring 提供了两种IoC 容器 : 

  - BeanFactory 
  - ApplictionContext

- BeanFactory:

  - 就像一个包含Bean 集合的工厂类。会在客户端要求时实例化 Bean 对象。

- ApplicationContext :  也被形容为 BeanFactory 的高级实现，在 BeanFactory 的基础上提供了一些额外的功能。

  - MessageSource : 管理message ,实现国际化
  - Application EventPublisher : 事件发布
  - ResourcePatternResolver : 多资源加载
  - EnvironmentCapable : 系统 Environment（profile + properties）相关
  - LifeCycle  : 管理生命周期
  - Closable : 关闭，释放资源
  - InitializingBean ： 自定义初始化
  - BeanNameAware : 设置 beanName 的 Aware 接口

- 常用的BeanFactory 容器： 

  - ClassPathApplicationContext : 从ClassPath 的 XML配置文件中读取上下文，并生成上下文定义 ：
    - ApplicationContext context = new ClassPathXmlApplicationContext(“bean.xml”);
  - FileSystemXmlApplication : 由文件系统中的XML配置文件读取上下文 : 
    - ApplicationContext context = new FileSystemXmlApplicationContext(“bean.xml”);
  - XmlWeApplicationContext : 有WEB 应用的 XML 文件读取上下文，例如在SpringMVC 中的使用情况。

  ### SpringBean 的生命周期

  - 实例化 Bean 对象 ：
    - Spring容器根据配置中的 Bean Definition **实例化** ***Bean*** 对象
      - BeanDefinition 可以通过 xml,Java 注解或者 JavaConfig 提供
    - Spring 使用依赖注入 ***填充***  所有属性，如Bean 中所定义的配置。
  - Aware 相关属性，注入到 ***Bean对象***  
    - 如果 **Bean** 实现了  ***BeanNameAware*** 接口，工厂通过传递的Bean 的BeanName  调用 setNeanName(String beanName) 方法。
    - 如果 **Bean** 实现了  ***BeanFactoryAware***  接口，工厂通过传递自身的实例来调用  setBeanFactory(BeanFactory beanFactory)方法。
  - 调用相应的方法，进一步初始化  Bean  对象...

  - 什么叫延迟加载

  - 默认情况下，容器启动之后会将所有作用域为 **单例** 的Bean都创建好，但是出于业务场景的考虑，并不是所有的单例Bean都需要

    - 在一开始就创建好，因此可以对不需要在容器启动后立刻创建的  Bean  设置 lazy-init 属性
      - 这样，当容器启动之后，作用域为单例的  bean  不会再立刻创建
      - 而是在获得该Bean (getBean(beanName/beanType)) 的时候进行创建

    ## Spring 注解

    - 当使用@Configration 和 @Bean  注解标注组件类来代替xml文件声明 BeanDefinition

      - @Bean ： 与xml 文件中 <Bean> 标签作用相同
      - @Configuration : 允许简单地通过调用  ***同一个类***  中的其他Bean 方法来定义 Bean 之间的依赖关系

      >

    - 如何在Spring中启动自动装配注解 ：  配置  <context : annotation-config>
    - @Component : 将 Java 类标记为 Bean . 它是任何Spring 管理组建的通用构造型
    - @Controller : 它将类标记为一个 Spring Web MVC 的前端控制器
    - @Service  : 和 @Component  类似，只是更好地指明了类的意图
    - @Repository :   @Component 注解的特化，为 **DAO** 提供了额外的好处， 它将DAO 导入 IOC 容器，并将未检查异常有资格转换为 Spring DataAccessException 
    - @Required :  用在 Bean 属性的 setter  方法，要求实例化 Bean  这个属性是必须的，否则会抛出 BeanInitializaiotnException
    - @Autowired : 可以更准确地控制在何处以及如何进行自动装配
      - 可以注解在 setter,构造方法，域变量，函数参数等自动装配Bean
      - 默认情况下它是类型驱动的注入

    - @Qualifier : 当创建了多个相同类型的 Bean  对它进行自动装配时，可以使用此注解指定使用的  BeanId  。
